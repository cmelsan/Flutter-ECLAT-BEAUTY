Resumen Ejecutivo

Síntesis del proyecto, alcance y resultados obtenidos

El presente documento recoge de forma exhaustiva el proceso de diseño, desarrollo e implementación de la aplicación móvil ÉCLAT Beauty, resultado de la migración completa de la plataforma web —construida con Astro.js— hacia una aplicación nativa Flutter para Android e iOS.

La migración abarca la totalidad de las funcionalidades de la tienda online: catálogo de productos, carrito de compra, proceso de pago con Stripe, gestión de pedidos y devoluciones, sistema de cupones, reseñas, lista de deseos y un panel de administración completo. El backend Supabase existente fue reutilizado íntegramente sin modificar el schema de la base de datos.

La arquitectura adoptada, Clean Architecture con patrón Feature-first, garantiza escalabilidad, mantenibilidad y facilidad de pruebas. El código ha sido generado con herramientas como Freezed y Riverpod, siguiendo principios SOLID en toda la base de código.

Contexto y Objetivos del Proyecto

Motivación, alcance y requisitos de la migración

Contexto

ÉCLAT Beauty es una tienda online de cosmética premium. La plataforma web, desarrollada con Astro.js y Supabase, cubría la operativa de venta online pero carecía de presencia móvil, limitando su alcance ante un segmento de clientes que prefiere comprar desde el teléfono.

La decisión de migrar a Flutter responde a la necesidad de mantener una única base de código para Android e iOS, reduciendo costes de desarrollo y facilitando el mantenimiento, sin sacrificar rendimiento nativo.

Objetivos del proyecto

  • Desarrollar una aplicación nativa Android/iOS con paridad de funcionalidades respecto a la web
  • Reutilizar el backend Supabase existente sin modificar el schema de la base de datos
  • Implementar la identidad visual premium de la marca: paleta rosa-dorado, tipografía Playfair Display
  • Garantizar pagos seguros y fluidos mediante Stripe en entorno móvil (Payment Sheet nativa)
  • Soporte para usuarios invitados (carrito offline con Hive) y autenticados
  • Panel de administración completo integrado en la misma aplicación
  • Código modular, testeable y preparado para escalar a nuevas funcionalidades
  • Seguridad robusta: RLS en BD, guards de navegación y gestión de secretos via variables de entorno

Análisis Comparativo: Web vs Flutter

Diferencias técnicas clave entre la plataforma original y la migrada

Paradigma de programación

⬛ Web — Astro.js + TypeScript
• Páginas estáticas/SSR generadas en build
• Estado con nanostores (stores reactivos)
• Componentes .astro + React/Svelte
• Routing basado en sistema de archivos
• CSS con Tailwind (clases utilitarias)
• Datos directamente desde Supabase JS Client
✅ App — Flutter + Dart 3
• UI declarativa con widgets inmutables
• Estado reactivo con Riverpod (StateNotifier)
• Widgets reutilizables por composición
• Routing declarativo go_router con guards
• Estilos en ThemeData + AppColors
• Datos a través de Repository Pattern

Gestión de pagos

⬛ Web — Stripe.js Elements
• Formulario de tarjeta en iframe
• Redirect a página de Stripe Checkout
• Webhook externo para confirmar el pago
✅ App — flutter_stripe
• Payment Sheet nativa del sistema operativo
• Soporte Apple Pay y Google Pay integrado
• Sin salir de la aplicación en ningún momento

Persistencia local

⬛ Web — localStorage
• Carrito de invitado en localStorage del navegador
• Sesión gestionada por cookies de Supabase Auth
✅ App — Hive (NoSQL embebido)
• Carrito de invitado en base de datos Hive
• Al hacer login: merge automático via RPC
• Hive cifrado en entorno de producción

Decisión de arquitectura: Feature-first vs. capas global

CriterioPor capas (global)Feature-first ✅ Adoptada
EscalabilidadDifícil con >5 módulosCada feature es autocontenida
Trabajo en equipoConflictos frecuentes en gitDesarrollo independiente por feature
TestabilidadDependencias cruzadasFeature se testea de forma aislada

Complejidad inicialSencilla de arrancarMayor, pero mucho más mantenible

Justificación de las elecciones técnicas clave

🧊
Freezed
Genera automáticamente fromJson, toJson, copyWith, == y hashCode. Elimina un 60% del boilerplate y garantiza inmutabilidad.
⚡
Riverpod 2
Providers type-safe, sin contexto de árbol de widgets, combinables. Soporte nativo para async. Superior a Provider clásico para proyectos complejos.
🧭
go_router
Routing declarativo con soporte a deep links, guards en redirect y URLs bien formadas. Simplifica la gestión de autenticación.
🔀
Either<F, T>
Los errores son parte explícita del tipo de retorno. Imposible ignorar un caso de error. Alternativa funcional segura a las excepciones.
🗄️
Hive
Base de datos NoSQL embebida en Dart, rápida y sin dependencias nativas. Ideal para persistir el carrito offline sin SQLite.
♻️
Supabase reutilizado
Sin modificar el schema PostgreSQL. La app Flutter consume exactamente los mismos datos que la web, garantizando consistencia total.

@freezed
abstract class Order with _$Order {
  const factory Order({
    required String id,
    @JsonKey(name: 'order_number') required String orderNumber,
    @JsonKey(name: 'user_id')        String? userId,
    @JsonKey(name: 'guest_email')    String? guestEmail,

    @JsonKey(name: 'created_at')     DateTime? createdAt,
    @JsonKey(name: 'delivered_at')   DateTime? deliveredAt,
    @JsonKey(name: 'return_deadline_at') DateTime? returnDeadlineAt,
    @JsonKey(name: 'return_reason')  String? returnReason,
    @JsonKey(name: 'order_items') @Default([]) List<OrderItem> orderItems,

  factory Order.fromJson(Map<String, dynamic> json) => _$OrderFromJson(json);

Gestión segura de secretos

Seguridad: El archivo .env con las API keys reales nunca se sube a Git. Está listado en .gitignore. Solo se incluye .env.example como plantilla de referencia para el equipo. 
# .env.example — Plantilla. Nunca incluir valores reales en el repositorio
SUPABASE_URL=https://xxxxxxxxxxxx.supabase.co
SUPABASE_ANON_KEY=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
STRIPE_PUBLISHABLE_KEY=pk_live_...
CLOUDINARY_CLOUD_NAME=nombre_del_cloud
SITE_URL=https://eclatbeauty.com

Capas de seguridad aplicadas

  • RLS en Supabase: cada fila tiene restricciones a nivel de base de datos, independientemente de la app cliente
  • Guards en go_router: el router verifica autenticación en cada cambio de ruta
  • Verificación doble de admin: guard de ruta + verificación en AdminLoginScreen en tiempo de ejecución
  • Variables de entorno: claves API fuera del código fuente con flutter_dotenv
  • Either<Failure, T>: errores propagados explícitamente, nunca silenciados
  • HTTPS obligatorio: toda comunicación con Supabase y Stripe usa TLS

Dificultades y Soluciones Adoptadas

Problemas técnicos encontrados y cómo se resolvieron

Incompatibilidades con Supabase Flutter 2.8.4

Al actualizar a la versión 2.8.4 del cliente Supabase para Flutter, se detectaron 58 errores de compilación por cambios de API. Se resolvieron sistemáticamente:

ProblemaAntesSolución (v2.8.4)
FetchOptions eliminado.select('id', const FetchOptions(count: CountOption.exact)).select('*').count(CountOption.exact)
count nullableresponse.count ?? 0response.count (ya no es nullable)
Método in_() renombrado.in_('status', [...]).inFilter('status', [...])
Tipo de query encadenadoPostgrestFilterBuilder querydynamic query
Casts innecesariosreturn response as Map<...>return response (tipo inferido)
Resultado: 0 errores de compilación tras aplicar los 5 tipos de correcciones en los 11 DataSources de la aplicación.

Merge atómico del carrito invitado al autenticarse

La migración del carrito de invitado al hacer login requería que el merge fuera atómico y manejara correctamente los duplicados (mismo producto en ambos carritos). La solución fue delegar la lógica completa en la función RPC migrate_guest_cart_to_user(p_session_id, p_user_id) de PostgreSQL, que ejecuta todo en una transacción única. Si el producto ya existe en el carrito del usuario, se suman las cantidades. El fallo en la migración no bloquea el login (error no crítico).

Concurrencia en cupones y stock

Dos usuarios comprando simultáneamente el mismo producto o aplicando el mismo cupón podían generar condiciones de carrera. La solución fue implementar funciones PostgreSQL con SELECT FOR UPDATE (row-level locking):

  • increment_coupon_usage_atomic(): bloquea la fila del cupón antes de incrementar el contador
  • create_order(): verifica y descuenta stock en transacción única, con rollback automático si el stock es insuficiente

Conclusiones y Trabajo Futuro

Evaluación del proyecto y próximas etapas de desarrollo

Conclusiones

La migración de ÉCLAT Beauty a Flutter ha sido completada satisfactoriamente. La aplicación implementa la totalidad de las funcionalidades de la plataforma web original, añadiendo además capacidades exclusivas del entorno móvil: carrito persistido offline, Payment Sheet nativa de Stripe con soporte a Apple Pay y Google Pay, y acceso desde cualquier dispositivo Android o iOS sin navegador.

La adopción de Clean Architecture con Feature-first ha demostrado ser acertada: los 12 módulos están perfectamente aislados, lo que facilita la incorporación de nuevas funcionalidades sin afectar al resto de la aplicación. El uso de Freezed, Riverpod y Dartz ha resultado en código expresivo, seguro y libre de boilerplate.

Los 0 errores de compilación, la integración limpia con Supabase, Stripe y Cloudinary, y la correcta implementación de seguridad a todos los niveles (RLS, guards, variables de entorno) confirman la solidez técnica de la solución entregada.
